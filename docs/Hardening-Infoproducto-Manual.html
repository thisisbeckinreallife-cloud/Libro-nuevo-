01 · Resumen ejecutivo

Qué cubre este manual y qué no.

Este documento sirve para llevar cualquier landing de un infoproducto digital al mismo nivel de hardening que arkwright.laralawn.com: legal al día con normativa española y europea, telemetría interna funcionando, self-service de recibo, y observabilidad del propio servidor — todo sin contratar servicios externos nuevos.

Lo que SÍ incluye

  • Sprint 1 · 4 páginas legales (aviso legal, privacidad, términos, cookies) y banner de consentimiento sin librerías externas.
  • Sprint 2 · Página de contacto + endpoint de salud + tracking de errores en BD + sitemap + robots + JSON-LD + OG image dinámica.
  • Sprint 3 · Stripe Customer Portal (recibo / método de pago / refund si decides activarlo) + tracking de conversión propio + dashboard /admin/analytics.

Lo que NO incluye (descartado por filtro)

  • Sentry / Rollbar / Bugsnag → sustituido por tabla ErrorLog + /admin/errors.
  • Plausible / Google Analytics / Mixpanel → sustituido por tabla Event + /admin/analytics.
  • UptimeRobot / Better Stack → sustituido por /api/health + curl/cron manual.
  • Hotjar / Microsoft Clarity → sin equivalente interno sencillo, queda fuera.
  • ConvertKit / Mailchimp para sequences → fuera; Resend ya envía transaccionales pero sin automation built-in.
Filtro central del cliente: nada que requiera abrir una cuenta nueva en otro SaaS. Todo lo que se decide aquí cabe en Postgres y se administra desde /admin.
02 · Decisiones de diseño

Por qué este manual mete todo dentro del propio Next.

Telemetría 100 % en Postgres

Cada visita y cada error vive en la misma base de datos que las compras. Ventaja: nunca pagas un plan por exportar tus datos, y los joins entre "evento" y "compra" son SQL directo. Desventaja: no hay UI de tableau-style out of the box — pero /admin/analytics ya cubre el 90 % del uso real (visits / día, embudo, top UTMs, top paths).

Cookie banner sin librería externa

El componente CookieBanner.tsx son ≈140 líneas, persiste el consentimiento en localStorage.cookieConsent versionado y con caducidad de 6 meses (cumple la guía de la AEPD). Expone window.__cookieConsent para que cualquier script de tracking lea el flag analytics antes de disparar.

Política española estricta por defecto

  • Aviso legal con datos LSSI-CE obligatorios (razón social, NIF, domicilio, registro mercantil).
  • Privacidad con tabla finalidad / base jurídica / conservación (RGPD UE 2016/679 + LOPDGDD 3/2018).
  • Términos con waiver explícito del derecho de desistimiento (Art. 103.m RD-Ley 1/2007) — necesario para "no refunds" en producto digital con acceso inmediato.
  • Política de cookies cumpliendo Art. 22.2 LSSI-CE.

JSON-LD Schema.org para AI search

Un solo componente JsonLd.tsx con dos exports: JsonLdGlobal (Organization + WebSite, inyectado en layout.tsx) y JsonLdLanding (Person + Book + Product con aggregateRating). Hoy ya posiciona en Rich Results. Mañana — cuando los LLMs sigan citando — sigue siendo la mejor inversión de 1 hora del proyecto.

Regla de oro: si una funcionalidad se puede resolver con Postgres + Next, vale más que un SaaS, aunque el SaaS tenga mejor UI. El coste no es la subscripción mensual — es la fragilidad de un dato que ya no controlas.

04 · Modelo de datos

Dos tablas nuevas en Prisma. Nada más.

Estas dos tablas se añaden al schema.prisma existente. La migración se genera con npx prisma migrate dev --name add_error_log_and_event.

Tabla ErrorLog

model ErrorLog {
  id        String   @id @default(uuid()) @db.Uuid
  source    String   // "checkout" | "webhook" | "biblioteca" | ...
  message   String
  stack     String?
  path      String?
  userAgent String?  @map("user_agent")
  ipHash    String?  @map("ip_hash")
  meta      Json?
  createdAt DateTime @default(now()) @map("created_at")

  @@index([createdAt])
  @@index([source, createdAt])
  @@map("error_logs")
}

Tabla Event

model Event {
  id           String   @id @default(uuid()) @db.Uuid
  kind         String
  path         String?
  sessionId    String?  @map("session_id")
  utmSource    String?  @map("utm_source")
  utmMedium    String?  @map("utm_medium")
  utmCampaign  String?  @map("utm_campaign")
  referrerHost String?  @map("referrer_host")
  abVariant    String?  @map("ab_variant")
  meta         Json?
  createdAt    DateTime @default(now()) @map("created_at")

  @@index([createdAt])
  @@index([kind, createdAt])
  @@map("events")
}

Helpers asociados

  • src/lib/logError.ts → logError({ source, error, path, userAgent, ip, meta }) persiste en error_logs. Best-effort: si falla la escritura, hace console.error y traga la excepción.
  • src/lib/events.ts → recordEvent({ kind, path, sessionId, utm*, meta }) persiste en events. Mismo patrón best-effort.
Ambas tablas son append-only. Para purgar, DELETE FROM error_logs WHERE created_at < NOW() - INTERVAL '90 days' mensual, o un cron en Railway.

Salud del sistema

/api/health devuelve 200 con un objeto si Prisma responde a SELECT 1, Stripe tiene secret key configurada y Resend tiene API key. Devuelve 503 con el detalle del fallo en cualquier otro caso. Para chequear desde fuera: 

curl -s https://arkwright.laralawn.com/api/health | jq

Si quieres notificación, programa un cron en Railway o un workflow de GitHub Actions que llame al endpoint cada 5 min y abra issue al fallar.

OG image dinámica

Generada con next/og + ImageResponse. Sin dependencias adicionales. Se sirve en /opengraph-image y se enlaza automáticamente por Next al declarar el archivo opengraph-image.tsx en App Router.

JSON-LD inyectado

  • JsonLdGlobal en layout.tsx · Organization + WebSite. Cubre cualquier URL.
  • JsonLdLanding en el componente de la landing · Person (autor) + Book + Product + aggregateRating.
Verifica antes de publicar: Google Rich Results Test (search.google.com/test/rich-results) acepta la URL de desarrollo o producción y te dice qué tipos detecta. Si te marca un fallo en Product, suele ser porque falta offers.priceCurrency o offers.availability.
07 · Sprint 3 · 1 día

Conversión y self-service.

Stripe Customer Portal

El endpoint POST /api/stripe/portal recibe { token } con el accessToken del Purchase (el mismo de /biblioteca), obtiene la sesión de Stripe, recupera (o crea si fue invitado) el Customer, abre una sesión temporal del portal y devuelve { url }. El front en BibliotecaHero.tsx tiene un botón "Ver mi recibo" que llama al endpoint y redirige.

Toggle a revisar: Stripe Dashboard → Settings → Customer Portal → "Allow customers to request refunds". Si tu política es "no refunds", desmárcalo. Si decides activarlo, el webhook ya escucha charge.refunded y marca Purchase.status = "refunded".

Tracking de conversión interno

Cinco kinds de evento, validados server-side en /api/track:

KindCuándo se disparaSubjeto a consent
page_loadCada cambio de pathname client-side.Sí
cta_clickClick en [data-track-cta] que NO sea checkout.Sí
checkout_startClick en [data-track-cta="checkout"].Sí
purchase_completedWebhook checkout.session.completed.No (legítimo interés)
refund_requestedWebhook charge.refunded.No

Garantías del tracker

  • El componente Tracker NUNCA registra si window.__cookieConsent.analytics !== true.
  • Usa navigator.sendBeacon con fallback a fetch keepalive. Best-effort: nunca espera respuesta.
  • Genera sessionId con crypto.randomUUID() en sessionStorage. Caduca al cerrar pestaña — no identifica personas.
  • UTM y referrer host se sniffean una vez por sesión y persisten.
  • Rate-limit silencioso en el servidor: 60 eventos/min por IP. Por encima, devuelve 200 sin grabar.

Cómo etiquetar tus CTAs

En cualquier botón / link de conversión, añade data-track-cta="label". El tracker hace event delegation con closest(), no hace falta pasar callbacks. 

<button data-track-cta="checkout"> Comprar </button>
<a href="/oferta" data-track-cta="hero-link"> Más info </a>

Dashboard /admin/analytics

Las métricas que ya devuelve sin tocar nada:

  • Big numbers · page_load · cta_click · checkout_start · purchase_completed (+ refund_requested si hay).
  • Embudo · ratio visit → checkout y checkout → purchase.
  • Serie diaria de visitas (últimos 30 días) como bar chart CSS.
  • Top paths, top utm_source / medium / campaign, top referrer hosts.
  • Breakdown por abVariant si lo usas para A/B test.

  • Modificar webhook para registrar purchase_completed y refund_requested.
  • /admin/analytics renderiza correctamente con datos reales.

Fase 5 · Verificación end-to-end

  • Compra de prueba con tarjeta Stripe 4242 4242 4242 4242.
  • Email transaccional recibido vía Resend.
  • Acceso a /biblioteca?token=….

  • Evento purchase_completed aparece en /admin/analytics.

sitemap.ts skeleton

import type { MetadataRoute } from "next";

const BASE = (process.env.NEXT_PUBLIC_SITE_URL ?? "")
  .replace(/\/$/, "");

export default function sitemap(): MetadataRoute.Sitemap {
  const now = new Date();
  const urls = [
    "", "/oferta", "/contacto",
    "/legal/aviso-legal", "/legal/privacidad",
    "/legal/terminos", "/legal/cookies",
  ];
  return urls.map((u) => ({
    url: `${BASE}${u}`,
    lastModified: now,
    changeFrequency: "monthly",
    priority: u === "" ? 1.0 : 0.7,
  }));
}

SQL útiles · purga y debug

-- Purgar errores de más de 90 días
DELETE FROM error_logs WHERE created_at < NOW() - INTERVAL '90 days';

-- Conversion rate últimos 30 días
SELECT
  COUNT(*) FILTER (WHERE kind = 'page_load')         AS visits,
  COUNT(*) FILTER (WHERE kind = 'checkout_start')    AS checkouts,
  COUNT(*) FILTER (WHERE kind = 'purchase_completed') AS purchases
FROM events
WHERE created_at >= NOW() - INTERVAL '30 days';

-- Top UTM source / campaign
SELECT utm_source, utm_campaign, COUNT(*) AS total
FROM events
WHERE created_at >= NOW() - INTERVAL '30 days'
  AND utm_source IS NOT NULL
GROUP BY 1, 2
ORDER BY total DESC
LIMIT 20;
